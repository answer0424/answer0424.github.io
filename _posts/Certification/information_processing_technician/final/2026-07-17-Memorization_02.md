## 애플리케이션 테스트의 분류

#### 프로그램 실행 여부에 따른 테스트

- 정적 테스트
  - 프로그램을 실행하지 않고 명세서나 소스 코드를 대상으로 분석하는 테스트
  - 종류: 워크 스루, 인스펙션, 코드 인스펙션
- 동적 테스트
  - 프로그램을 실행하여 오류를 찾는 테스트
  - 종류: 블랙박스 테스트, 화이트박스 테스트

<br>

#### 테스트 기반에 따른 테스트

- 명세 기반 테스트
- 구조 기반 테스트
- 경험 기반 테스트

<br>

#### 시각에 따른 테스트

- 검증(Verification) 테스트: 개발자의 시각
- 확인(Validation) 테스트: 사용자(고객)의 시각

## 테스트 기법에 따른 애플리케이션 테스트

#### 화이트박스 테스트의 종류

- 기초 경로 검사: 절차적 설계의 논리적 복잡성을 측정할 수 있게 해줌
- 제어 구조 검사: 조건 검사, 루프 검사, 데이터 흐름 검사

<br>

#### 화이트박스 테스트의 검증 기준

- 문장 검증 기준
- 분기 검증 기준
- 조건 검증 기준
- 분기/조건 기준

<br>

#### 블랙박스 테스트의 종류

- 동치 분할 검사
- 경계값 분석
- 원인-효과 그래프 검사
- 오류 예측 검사
- 비교 검사

## 통합테스트

#### 하향식 통합 테스트

1. 주요 제어 모듈의 종속 모듈은 스텁으로 대체
2. 깊이 우선 또는 넓이 우선 등의 통합 방식에 따라 하위 모듈인 스텁들이 한번에 하나씩 실제 모듈로 교체
3. 모듈이 통합될 때마다 테스트
4. 새로운 오류가 발생하지 않음을 보증하기 위해 회귀 테스트를 실시

<br>

#### 상향식 통합 테스트

1. 하위 모듈들을 클러스터로 결합
2. 상위 모듈에서 데이터의 입출력을 확인하기 위해 더미 모듈인 드라이버를 작성
3. 통합된 클러스터 단위로 테스트
4. 테스트가 완료되면 클러스터는 프로그램 구조의 상위로 이동하여 결합하고 드라이버는 실제 모듈로 대체
