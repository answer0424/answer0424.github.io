## 핵심개념

- Working Directory: 실제 작업 중인 파일
- Staging Area: 커밋할 파일을 임시로 올려두는 공간
- Local Repository: 내 PC에 저장된 Git 이력
- Remote Repository: Github, GitLab 같은 원격 저장소

---

## Branch 개념

> 브랜치는 작업 공간을 분리하는 기능

---

## Git Flow

> 대표적인 브랜치 전략

```text
main
 └── develop
      ├── feature/login
      ├── feature/order
      └── release/1.0.0
main
 └── hotfix/payment-error
```

<br>

#### 흐름

- 기능 개발

> develop -> feature 생성 -> 개발 완료 -> develop 병합

- 배포 준비

> develop -> release 생성 -> 테스트 -> main 병합 -> develop에도 병합

- 운영 긴급 수정

> main -> hotfix 생성 -> 수정 -> main 병합 -> develop에도 병합


<br>

#### 장점

- 대규모 프로젝트에 적합
- 운영, 개발, 배포 준비가 명확히 분리됨

<br>

#### 단점

- 브랜치가 많고 복잡
- 작은 팀이나 빠른 배포 환경에서는 무거울 수 있음

---

## Github Flow

```text
main
 └── feature/login
```

<br>

#### 흐름

```text
main에서 브랜치 생성
→ 작업
→ Pull Request
→ 리뷰
→ main 병합
→ 배포
```

<br>

#### 적합한 경우

- 웹 서비스처럼 자주 배포하는 프로젝트
- CI/CD가 잘 구성된 팀
- 작은 팀

<br>

#### 장점

- 단순
- 브랜치 관리가 쉬움

<br>

#### 단점

- 배포 전 별도 release 관리가 약함

---

## Merge와 Rebase


#### Merge

- 브랜치를 합치는 가장 기본 방식
- 특징
  - 커밋 이력이 그대로 남음
  - 병합 커밋이 생길 수 있음
  - 안전함

<br>

#### Rebase

- 내 브랜치의 시작점을 최신 브랜치 뒤로 옮김
- 특징
  - 커밋 이력이 깔끔해짐
  - 잘못 쓰면 위험함
  - 공유 브랜치에서는 조심해야 함

---

## Reset과 Revert의 차이

#### Reset

- 커밋 이력을 되돌림

```bash
# 커밋만 취소 파일 변경은 유지
git reset --soft HEAD~1
```


```bash
# 커밋 취소 + staging 취소, 파일 변경은 유지
git reset --mixed HEAD~1
```


```bash
# 커밋 취소 + 파일 변경까지 삭제
git reset --hard HEAD~1
```

<br>

#### Revert

- 기존 커밋을 취소하는 새 커밋을 만듦

```bash
git revert 커밋ID
```

## stash

- 작어 중인 내용을 임시 저장
- 사용 상황

  - pull 받아야 하는데 현재 작업물이 걸릴 때
